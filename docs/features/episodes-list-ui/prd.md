# 에피소드 목록 UI (episodes-list-ui) PRD

이 문서는 에피소드 목록 대시보드 화면 및 그리드 컴포넌트를 설계하고 개발하기 위한 단일 기준 문서(PRD)이다.

---

## 1. 개요

임포트 완료된 에피소드 카드를 표시하고, 진행 중인 에피소드의 상태를 실시간 폴링하며, 실패한 카드는 손쉽게 재시도할 수 있는 학습자 중심의 대시보드 홈 UI를 구축한다.

---

## 2. 사용자 스토리

- **학습자로서**, 앱에 접속했을 때 내가 임포트하여 학습하고 있는 에피소드 카드를 직관적으로 보고 즉시 학습 상세 페이지로 진입하고 싶다.
- **학습자로서**, 현재 다운로드나 자막 정합이 진행 중인 에피소드의 진행 상태를 홈 화면을 떠나지 않고도 프로그레스 바를 통해 실시간으로 확인하고 싶다.
- **학습자로서**, 잘못 다운로드받았거나 더 이상 공부하지 않는 에피소드의 쓰레기통 버튼을 눌렀을 때, 확인 팝업을 거쳐 안전하게 데이터를 지우고 싶다.

---

## 3. 아키텍처 3개 시나리오 제안 및 비교

에피소드 목록 데이터 조회, 진행 상태 조건부 폴링 및 다이얼로그 제어를 구현하기 위한 3가지 대안을 제시합니다.

### 안 A — React Client Component + `useEffect`/`fetch` 직접 구현 (추천)

`src/app/page.tsx`는 Next.js 라우트 진입점이 되고, 그 하위의 실질적 화면인 대시보드를 Client Component로 선언하여 React 표준 `useState`, `useEffect`, `fetch` 및 `setInterval`을 조합해 데이터를 조회하고 조건부 폴링을 제어하는 방식.

### 안 B — Client Component + SWR 라이브러리 도입

Next.js 클라이언트 레이어에 `swr` 라이브러리를 의존성에 추가하여, 선언적 데이터 캐싱 및 `refreshInterval` 설정을 활용해 조건부 폴링과 에피소드 데이터 상태 관리를 처리하는 방식.

### 안 C — Server Component (RSC) + 클라이언트 전용 Poller (`router.refresh()`)

기본 목록 렌더링은 Next.js Server Component인 `app/page.tsx`에서 파일 데이터 로드를 직접 수행하고, 진행 중인 에피소드가 1개 이상 감지되면 클라이언트 Poller 컴포넌트를 주입하여 3초 주기로 `router.refresh()`를 수행해 서버 단에서 데이터를 리로드하게 하는 방식.

## 4. Out of Scope

이번 `episodes-list-ui` 피처에서 **구현하지 않는 비범위 사항**입니다:

1. **에피소드 검색 및 다중 필터링**: 에피소드가 수십 개 이상 쌓였을 때를 대비한 검색 바(Search Bar), 태그 필터 등은 MVP 이후 단계로 배제합니다.
2. **드래그 앤 드롭 정렬 (Reordering)**: 카드 그리드 상에서 사용자가 마우스로 드래그하여 순서를 커스텀 배치하는 기능은 배제합니다.
3. **대본 미리보기(Preview) 레이어**: 홈 화면 카드 내에서 대본의 내용 일부를 미리 파싱해 보여주는 UI는 구현 범위에서 제외합니다.
4. **모바일 터치 스와이프 제스처 삭제**: 모바일 화면 대응은 CSS 반응형만 준수하며, 터치 스와이프를 통한 삭제 등의 복잡한 모바일 전용 인터랙션은 제외합니다.

## 5. 기술 결정 (ADR-003)

### ADR-003: React Client Component + useEffect/fetch 기반 조건부 폴링 직접 구현

**Status**: Accepted (2026-06-29)

**Context**

- 홈 대시보드 화면은 백그라운드 잡(임포트 진행 상황)의 진행률을 3초 주기로 정확히 렌더링해야 합니다.
- 또한 에피소드 카드 개별 삭제 시 사용자 실수 방지를 위한 다이얼로그 노출, 실패 에피소드의 `/import?videoId=` 이동 등 활발한 클라이언트 인터랙션이 발생합니다.

**Decision**

- Next.js App Router의 `src/app/page.tsx`는 서버 진입점 역할만 담당하고, 실질적인 대시보드 구조는 클라이언트 컴포넌트(`src/components/episode/EpisodeDashboard.tsx`)로 위임합니다.
- React 내장 `useState`, `useEffect` 및 `fetch`를 이용하여 `/api/episodes` 데이터를 관리하고, 진행 중인 에피소드가 존재할 때만 `setInterval`을 활성화하여 조건부 폴링을 수행하도록 설계합니다.

**Alternatives**

- **안 B (SWR 라이브러리 도입)** — 거부. SWR의 선언적 훅은 편리하지만 신규 종속성을 추가하여 package-lock 빌드 충돌 우려를 증가시키고, 기존에 작성된 `/import` 페이지의 React 상태 폴링 패턴과 통일성을 깨뜨리므로 채택하지 않았습니다.
- **안 C (Server Component + router.refresh)** — 거부. 3초마다 페이지 전체의 RSC 페이로드를 다시 전송받는 방식은 로컬 환경이라 하더라도 지연 및 리소스 낭비가 심하며, 브라우저 스크롤 위치나 클릭 포커스 상태가 유실되는 부작용이 있어 기각했습니다.

**Consequences**

- **장점 (+)**: React 표준 API만을 사용하여 의존성이 고립되며 깨끗한 번들 구조를 유지합니다. 기존 `/import` 화면에서 이미 검증 완료된 `useImportStatus` 훅 등의 설계 형태와 정합성을 지켜 유지보수 효율이 매우 높습니다.
- **단점/한계 (-)**: 타이머 메모리 해제(Clean-up) 처리를 정밀하게 코딩하지 않으면 성능 누수가 발생할 수 있습니다. 이는 Vitest 단위 테스트에서 타이머 모킹(`vi.useFakeTimers()`)을 통해 리프레시와 타이머 클린업의 라이프사이클을 완벽히 검증할 것입니다.
