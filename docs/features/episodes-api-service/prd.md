# 에피소드 및 오디오 API 서비스 (episodes-api-service) PRD

이 문서는 에피소드 CRUD API와 HTTP Range 지원 오디오 스트리밍 API를 구축하기 위한 단일 기준 문서(PRD)이다.

---

## 1. 개요

파일 시스템에 임포트된 에피소드 목록을 홈 화면에 표시하고, 불필요한 에피소드를 안전하게 삭제하며, 플레이어에서 끊김 없는 오디오 구간 탐색(Seeking) 및 부분 스트리밍을 제공하는 백엔드 API 레이어를 구축한다.

---

## 2. 사용자 스토리

- **사용자로서**, 홈 화면에 접속했을 때 현재 임포트 완료된 에피소드뿐만 아니라 다운로드 중인 항목 및 실패한 항목까지 일목요연하게 보고 싶다.
- **사용자로서**, 불필요한 에피소드 삭제 버튼을 눌렀을 때, 현재 다운로드 중인 상태가 아니라면 안전하게 모든 데이터 파일들이 깨끗이 지워지길 바란다.
- **사용자로서**, 오디오 플레이어에서 재생 바를 특정 구간으로 이동(Seek)시켰을 때, 대기 지연 없이 해당 지점부터 오디오가 부드럽게 재생되길 바란다.

---

## 3. 아키텍처 3개 시나리오 제안 및 비교

HTTP Range 지원 오디오 스트리밍 및 에피소드 삭제/조회 기능을 구현하기 위한 3가지 기술적 대안을 제시합니다.

### 안 A — Next.js Route Handler + Node fs/stream (직접 구현)

`app/api/episodes/[id]/audio/route.ts` 경로에서 Next.js Route Handler를 만들고, 들어오는 `Range` 헤더를 직접 파싱하여 `fs.createReadStream`과 Node/Web Stream API를 조합해 `206 Partial Content` 응답을 처리하는 방식.

### 안 B — public 디렉토리 심볼릭 링크 활용 (정적 서빙 위임)

로컬 데이터 저장소(`.shadowing/episodes`)를 Next.js의 정적 서빙 경로인 `public/episodes` 하위로 심볼릭 링크 처리하여, 웹 서버(Next.js 내장 정적 서빙)가 Range 요청 처리를 전담하게 하고 API 레이어는 삭제/조회만 담당하는 방식.

### 안 C — 외부 스트리밍 패키지(예: `send` 등) 래핑 활용

Node.js 진영에서 검증된 정적 파일/Range 처리 스트리밍 라이브러리(예: `send`, `range-parser`)를 추가적으로 설치하여 Next.js Route Handler 내부에서 Node raw Response 객체를 통해 응답을 래핑 처리하는 방식.

## 4. Out of Scope

이번 `episodes-api-service` 피처에서 **구현하지 않는 비범위 사항**입니다:

1. **대본 및 자막 파일 전용 스트리밍 API**: 대본(`.txt`)과 자막(`.vtt`) 리소스는 별도 스트리밍 API를 만들지 않고, 기존 `episodes.ts` 서비스 레이어를 통해 직접 파일을 읽어 파싱하거나 별도의 정적 파일 서빙 없이 클라이언트에 필요한 JSON 객체로 전달하여 처리합니다.
2. **에피소드 정보 수정 (Update) API 및 화면**: 제목 변경, 태그 추가 등의 메타데이터 수정 기능은 이번 범위에서 제외합니다.
3. **일괄 삭제 (Batch Delete) API**: 한 번의 요청으로 여러 에피소드를 동시에 지우는 API는 제공하지 않으며, 오직 단일 에피소드 ID 삭제만 지원합니다.
4. **스트리밍 대역폭 제한 (Rate Limiting)**: 단일 사용자 로컬 환경을 전제로 하므로 성능 최적화를 위한 별도 대역폭 제어나 동시성 제한 장치는 두지 않습니다.

## 5. 기술 결정 (ADR-002)

### ADR-002: Next.js Route Handler + Node fs 기반 HTTP Range 스트리밍 직접 구현

**Status**: Accepted (2026-06-29)

**Context**

- 미디어(mp3) 재생 시 사용자가 임의의 구간으로 버퍼링 없이 즉시 탐색(Seek)하기 위해서는 HTTP `Range` 요청을 정상 처리해주어야 합니다.
- 임포트 진행 중(`downloading`)인 불완전한 상태에서는 재생 요청을 안전하게 차단(`409 Conflict`)해야 하며, 파일이 아예 누락된 경우에는 클라이언트에 명확한 에러(`404 Not Found`)를 반환해 플레이어 UI를 예외 처리해야 합니다.

**Decision**

- Next.js App Router의 `src/app/api/episodes/[id]/audio/route.ts`에 Route Handler를 생성하고, 들어오는 HTTP `Range` 헤더를 직접 파싱하여 `fs.createReadStream`과 Web Stream을 조합해 `206 Partial Content` 청크 스트리밍 응답을 처리합니다.

**Alternatives**

- **안 B (public 심볼릭 링크 위임)** — 거부. 배포/개발 빌드 시점에 로컬 영속 공간을 public 디렉토리와 연결하기 위한 심볼릭 링크 작업 스크립트가 추가되어 복잡해지고, 다운로드 도중 재생 요청에 대해 `409 Conflict` 등의 예외 규칙을 인터셉트하여 제어하기가 까다로워 채택하지 않았습니다.
- **안 C (외부 패키지 래핑 활용)** — 거부. `send` 등 Node.js 기반 스트리밍 라이브러리는 Next.js App Router의 Web Standard `Response` 구조와 연동하려면 접착 어댑터 코드가 길어져 복잡해지며, 굳이 신규 패키지 의존성을 추가하여 package-lock을 불안정하게 만들 이유가 없어 거부했습니다.

**Consequences**

- **장점 (+)**: 외부 서드파티 종속성 추가 없이 오직 Node.js 표준 모듈만으로 완벽히 고립된 구현이 가능합니다. 에피소드 메타데이터 상태와 결합하여 404/409 오류를 안전하게 분기할 수 있으며, Mock File System을 주입하여 결정론적인 단위/통합 테스트를 작성하기가 매우 용이합니다.
- **단점/한계 (-)**: HTTP 사양에 맞추어 byte-range 파싱, `Content-Range`, `Content-Length` 계산 등 규격 코드를 손수 정확하게 작성해야 하므로 버그 발생 가능성이 상대적으로 높습니다. 이는 `node:assert` 기반의 촘촘한 단위 테스트 시나리오(정상, 경계, 예외 범위) 검증을 통해 완벽히 방어할 것입니다.
