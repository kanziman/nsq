# Feature 기획 워크플로우 스킬

이 스킬은 신규 기능의 아이디어(spec-original.md)를 개발 가능한 구체적인 작업 단위(issues.md 및 GitHub Issues)로 변환하는 점진적 기획 파이프라인을 강제합니다. 각 단계 사이에는 사용자의 명시적인 확인과 승인을 받아야 하는 **승인 게이트([GATE])**가 존재합니다.

---

## 📌 전체 흐름도

```
docs/features/{name}/spec-original.md (초기 아이디어)
    ↓ [단계 1] 요구사항 인터뷰 진행
docs/features/{name}/spec-fixed.md (확정된 요구사항)
    ↓ [GATE 1] 사용자의 내용 및 용어 정의 확정 승인
    ↓ [단계 2] PRD 초안 작성 및 3가지 기술 아키텍처 시나리오 비교 제안
    ↓ [GATE 2] 사용자의 아키텍처(ADR) 시나리오 선택
    ↓ [단계 3] 선택된 안을 기반으로 상세 PRD 및 Out of Scope 기술
    ↓ [GATE 3] 사용자의 범위(Out of Scope) 및 최종 PRD 확정 승인
docs/features/{name}/prd.md (단일 기준 문서)
    ↓ [단계 4] 수직 슬라이싱 원칙에 따른 실행 가능한 이슈 분해
    ↓ [GATE 4] 사용자의 이슈 목록 및 수직 슬라이스 기준 충족 여부 승인
docs/features/{name}/issues.md + GitHub Issues 등록
```

---

## 1단계: 요구사항 인터뷰 (spec-fixed.md 작성)

### 목적

초기 기능 정의서의 모호함을 완전히 제거하고, 엣지 케이스 및 결정 사항을 확정합니다.

### 입력

- `docs/features/{name}/spec-original.md` (초기 기능 정의서)
- 프로젝트 코드베이스 (기존 구조, 파일 패턴, 스타일 표준 등 참조)

### 핵심 수행 지침

1.  **인터뷰어로 작동할 것**: 사용자의 초기 문서를 바탕으로 코드베이스를 분석하고, 구현 전에 결정해야 할 의문 사항들을 질문합니다.
2.  **질문 진행 규칙**:
    - **한 번에 단 하나의 질문만 던질 것**. 사용자가 답변하면 그 내용을 기록하고 다음 질문으로 넘어갑니다.
    - 질문을 던질 때 **추천하는 대안 방식과 그 이유**를 함께 기재하여 사용자가 합리적으로 판단할 수 있게 유도하십시오.
3.  **질문 대상 영역**:
    - 기능의 핵심 사용자(Primary User) 정의
    - 최소 동작 시나리오 3개 (기본 사용 흐름)
    - 데이터 저장 방식 및 기존 구조와의 유기적 관계
    - 경계 조건 (최대값, 빈 값, 중복, 동시성)
    - 예외 및 에러 처리 (실패 시 UI 상에 어떻게 전달하는가)
    - 기존 UI 컴포넌트 재사용 일관성
    - 성능 제약 (응답 시간, 데이터 크기 제한 등)
    - 향후 확장성 및 종속성
4.  **용어 정의 (Ubiquitous Language)**:
    - 코드, 문서, 대화에서 혼선 없이 동일하게 사용할 단어를 합의하고 정규화합니다.
    - 이 내용은 최종 `spec-fixed.md`에 "용어 정의" 섹션으로 명시해야 합니다.

### 산출물

- `docs/features/{name}/spec-fixed.md`

### [GATE 1] 승인 게이트

- **사용자가 `spec-fixed.md`의 내용과 정의된 용어들을 확인하고 승인하기 전까지 다음 단계로 넘어가지 말고 대기하십시오.**

---

## 2단계: PRD 작성 및 기술 결정 (ADR)

### 목적

요구사항, 디자인, 기술 의사결정, 범위를 하나의 `prd.md`로 통합하여 "단일 진실 원천(Single Source of Truth)"을 생성합니다.

### 입력

- `docs/features/{name}/spec-fixed.md`
- `docs/features/{name}/design.md` (존재할 경우 참조)
- 프로젝트 코드베이스

### 핵심 수행 지침

#### 2-1. PRD 기본 골격 구성

- **개요**: 기능의 비즈니스 목적 및 목표 범위
- **사용자 스토리**: 구체적인 사용자 관점의 가치 흐름
- **기술 결정**: 어떤 설계/아키텍처 위에 올릴 것인가 (아래에서 ADR 수행)
- **Out of Scope**: 구현 범위에서 명확히 배제할 사항
- **용어 정의**: `spec-fixed.md`에서 확정된 용어의 동기화

#### 2-2. 아키텍처 3개 시나리오 제안 및 비교

아래 **7가지 고정 기준**을 적용하여 최소 3가지 아키텍처 대안을 비교 분석해 표로 정리하고 제안하십시오.

1.  **데이터 구조**: 어떤 형태로 데이터를 저장/표현하고 통신하는가
2.  **API 레이어 변경지점**: 기존 백엔드/서버 라우트의 변경 범위
3.  **상태관리 변경지점**: Context, Store 등 클라이언트 상태관리 변경 범위
4.  **핵심 동작**: 가장 중요하고 까다로운 핵심 기능의 흐름
5.  **컴포넌트 구조**: 신규 생성/수정될 컴포넌트 구조 및 의존성
6.  **기존 패턴과의 일관성**: 기존 코드 관습 및 디자인 시스템 준수 여부
7.  **테스트 용이성**: 단위 테스트 및 E2E 테스트 구성 난이도

### [GATE 2] 아키텍처 선택 게이트

- **사용자가 비교 분석 표를 읽고 3가지 아키텍처 시나리오 중 하나를 선택할 때까지 작업을 멈추고 대기하십시오.**

#### 2-3. 선택된 안에 대한 ADR (Architecture Decision Record) 작성

PRD의 "기술 결정" 섹션에 아래 **ADR 4대 요소**를 빠짐없이 기술하십시오.

- **Context**: 이 결정이 필요한 배경과 맥락
- **Decision**: 최종적으로 선택한 방식과 구체적인 구조
- **Alternatives**: 기각된 대안 시나리오들과 **각 대안이 거부된 구체적인 이유** (이유 없는 거부 기재 금지)
- **Consequences**: 선택으로 발생하는 트레이드오프 (**장점뿐만 아니라 단점과 한계도 반드시 명시**)

#### 2-4. Out of Scope 명시

- 기능을 최소 단위(MVP)로 빠르게 동작시키기 위해 **이번에 절대로 구현하지 않을 기능**(예: 부가 편의성 검색, 추천 알고리즘, 세부 데코레이션 등)을 구체적으로 선언하십시오.
- 범위를 한정하지 않으면 범위 초과(Scope Creep) 구현이 일어나 코드 정합성이 떨어집니다.

### [GATE 3] 범위 및 PRD 승인 게이트

- **사용자가 최종 완성된 `prd.md`와 Out of Scope 범위를 확인하고 승인하기 전까지 다음 단계로 넘어가지 말고 대기하십시오.**

### 산출물

- `docs/features/{name}/prd.md`

---

## 3단계: 이슈 분해 (issues.md 및 GitHub Issues)

### 목적

PRD 명세를 바탕으로 실행 가능하고 테스트가 용이한 개발 태스크 단위로 쪼개어 작업 보드에 올립니다.

### 입력

- `docs/features/{name}/prd.md`

### 핵심 수행 지침

1.  **수직 슬라이싱 (Vertical Slicing) 원칙**:
    - **"이 이슈 하나만 완료하면 사용자에게 기능적으로 검증 가능한 독립적 동작이 나타나는가?"**를 자문하십시오.
    - **수평 슬라이싱(레이어별 분할: API 먼저 ➔ Context ➔ UI)은 엄격히 금지합니다.** 수평 분할은 마지막 단계까지 동작을 볼 수 없어 독립적인 TDD 피드백이 불가능합니다.
    - 하나의 이슈는 데이터 저장, 통신, 화면 렌더링까지 수직으로 관통하는 조각이어야 합니다.
2.  **이슈의 크기**:
    - 반나절에서 최대 하루 이내에 테스트 작성 및 구현(Red-Green-Refactor)이 완료될 수 있는 콤팩트한 크기여야 합니다.
3.  **의존성 순서**:
    - 선행 이슈가 완료되어야 후속 이슈를 정상 진행할 수 있도록 물 흐르듯 의존성 순서를 정렬하고 역방향 개발을 원천 차단하십시오.
4.  **Acceptance Criteria ( Given-When-Then 형식)**:
    - 각 이슈마다 명확한 승인 조건(AC)을 기재하십시오.
    - `Given [사전 조건], When [사용자 행동/입력], Then [기대 결과]` 문법을 준수해야 합니다.

### [GATE 4] 이슈 승인 및 등록 게이트

- **작성된 이슈 목록(`issues.md`)의 수직 슬라이싱 정합성, 의존성 순서, AC 적정성을 사용자가 승인하기 전까지 대기하십시오.**
- 사용자 승인이 완료되면 `gh CLI`를 이용하여 실제 레포지토리 이슈를 등록하고 칸반 보드의 `Todo` 컬럼에 카드를 추가합니다.

### CLI 실행 패턴 (등록용)

```bash
gh issue create --title "[Feature Name] 이슈 제목" --body "설명 및 Given-When-Then Acceptance Criteria"
gh project item-add <PROJECT_NUMBER> --owner <OWNER> --url <ISSUE_URL>
```

### 산출물

- `docs/features/{name}/issues.md`
- GitHub Issues 등록 완료 및 칸반 보드 갱신

## ⚠️ 승인 게이트 요약 (에이전트 동작 체크리스트)

에이전트는 사용자의 판단과 컨펌 없이 단계를 독단적으로 건너뛰어서는 안 됩니다.

| 검증 시점      | 게이트 이름 | 확인 및 컨펌받아야 할 내역                                     |
| :------------- | :---------- | :------------------------------------------------------------- |
| **1단계 후**   | `[GATE 1]`  | `spec-fixed.md` ➔ 모호성 해결 여부 및 용어 사전 합의 확인      |
| **2-2단계 후** | `[GATE 2]`  | `3가지 아키텍처 비교` ➔ 사용자가 개발할 시나리오 1개 선택 확인 |
| **2-4단계 후** | `[GATE 3]`  | `prd.md` ➔ Out of Scope 범위 및 최종 PRD 구조 합의 확인        |
| **3단계 후**   | `[GATE 4]`  | `issues.md` ➔ 수직 슬라이싱 기준 충족 및 AC 구조 컨펌 확인     |
