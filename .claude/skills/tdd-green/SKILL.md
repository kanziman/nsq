# TDD Green 구현 스킬 (tdd-green)

이 스킬은 실패 중인 테스트 코드들을 대상으로, 테스트를 통과시키는 데 꼭 필요한 **최소한의 구현 코드만 작성(Green)**하여 기능 명세를 완전히 완성하는 TDD 프로세스를 강제합니다.

## 사용법

- `/tdd-green [GitHub 이슈 번호]` (예: `/tdd-green 1`)

## 입력 인자

- `$ARGUMENTS`: GitHub 이슈 번호 (예: `1`)

## 상세 실행 단계

이 스킬이 활성화되면 다음 단계를 순서대로 엄격히 수행하며 피드백 루프를 반복합니다.

### 1단계: 실패 테스트 목록 확인

- **동작**: `npm test` (또는 프로젝트에 맞는 테스트 실행 명령어)를 실행하여 현재 실패하고 있는 테스트들의 전체 목록을 확보합니다.

### 2단계: 모듈 및 테스트 감지 검증 (Collect 실패 감지)

- **비교 분석**:
  - `docs/features/{tag}/issue-{번호}.md` 파일에 명시된 시나리오 체크박스(`- [ ]`)의 총 개수를 파악합니다.
  - Vitest 또는 테스트 러너가 보고하는 총 `Tests` 수와 비교합니다.
- **예외 복구**:
  - 두 개수가 서로 다른 경우(예: 테스트 러너가 테스트 파일을 아예 찾지 못하거나 일부를 누락한 경우), import 대상이 되는 구현체 모듈 파일이 아예 존재하지 않는 상태일 가능성이 큽니다.
  - 이 경우, 대상 모듈을 생성하고 1단계에서 확정된 **시그니처만 작성한 스텁(Stub) 파일**을 구현 경로에 생성하여, 테스트 러너가 전체 테스트 케이스들을 정상적으로 로드하고 인식할 수 있도록 조치합니다.

### 3단계: 디자인 시스템 컨텍스트 분석

- **동작**: 구현하려는 대상이 UI 컴포넌트일 경우 다음 디자인 명세를 먼저 분석합니다.
  - `docs/design-system`가 존재하면 로드하여 컴포넌트의 색상, 타이포그래피, 간격, 레이아웃, 그리고 상태별(Hover, Focus, Disabled 등) 시각 스펙을 파악합니다. (파일이 없으면 프로젝트 내 디자인 토큰이나 설정 문서를 찾아 참고합니다)
  - `docs/features/{feature}/issue-{번호}.md` 내에 "디자인 참고" 섹션이 있다면 해당 시각적 세부 사항도 함께 확인합니다.
  - `CLAUDE.md`에 스타일링 컨벤션 또는 디자인 토큰 관련 규칙이 적혀 있다면 이를 엄격히 준수합니다.
- **예외**: 관련 디자인 문서가 아예 존재하지 않는 로직 전용 모듈의 경우 이 단계를 생략하고 `CLAUDE.md` 컨벤션만 준수합니다.

### 4단계: 개별 테스트 통과를 위한 최소 구현

- **동작**: 실패하고 있는 첫 번째 테스트 케이스를 선정하고, 이를 **통과시키는 데 필요한 가장 단순하고 최소한의 코드**만 구현 파일에 작성합니다.
- **스타일 적용 규칙**:
  - UI 컴포넌트를 구현할 때는 테스트가 요구하는 검증 사항을 만족하는 동시에, 3단계에서 파악한 `design.md`에 명시된 Tailwind CSS 클래스, 레이아웃 구조, 상태별 표현 등을 온전히 코드로 녹여내야 합니다.
- **제약 사항**:
  - **테스트 코드가 명시적으로 검증하지 않는 기능이나 과도한 일반화, 불필요한 추상화는 절대로 구현하지 않습니다.**
  - `design.md` 등에 기재되지 않은 임의의 시각 요소를 독단적으로 컴포넌트에 추가해서는 안 됩니다.

### 5단계: 피드백 루프 (테스트 실행 및 수정)

- **동작**: 해당 테스트 파일을 대상으로 `npm test`를 실행하여 작성한 최소 구현 코드가 정상 통과하는지, 기존 테스트에 회귀 오류(Regression)를 일으키지 않는지 확인합니다.
- **테스트가 여전히 실패하는 경우**:
  - 에러 로그를 꼼꼼히 확인하고 실패의 원인을 식별합니다.
  - 코드를 수정한 뒤 다시 테스트 실행 과정을 거칩니다.
  - **이 피드백 루프는 최대 5회까지만 반복합니다.**
- **동작 중단 조건**:
  - 5회 이상 수정을 시도했음에도 테스트가 계속해서 실패할 경우, 독단적으로 더 수정하지 않고 개발자에게 실패 로그와 시도 내역을 즉시 보고한 뒤 단계를 중단합니다.

### 6단계: 시나리오 체크박스 업데이트

- 테스트가 정상적으로 통과하면, `docs/features/{feature}/issue-{번호}.md` 파일에서 방금 해결한 테스트 시나리오 항목의 체크박스를 `[x]`로 변경합니다.

### 7단계: 다음 테스트 반복 처리

- 여전히 실패하고 있는 다른 테스트 케이스가 남아 있다면, 4단계로 돌아가 전체 과정을 반복합니다.

### 8단계: 테스트 커버리지 측정 및 결과 검토

- 모든 테스트가 정상적으로 통과(`Green`)하면, 테스트 커버리지를 측정합니다.
  - 커맨드: `npx vitest run --coverage` (또는 프로젝트에 맞는 커버리지 도구 실행)
- **커버리지 분석**:
  - 커버되지 않은 라인(Uncovered Lines)이 존재하는 경우, 설계 당시 누락된 테스트 시나리오가 있는지 파악하여 개발자에게 상세히 보고합니다.
  - **단, 단순히 커버리지를 높이기 위해서 테스트 코드나 시나리오를 임의로 직접 추가하는 것은 금지되어 있습니다.** 커버리지 수치와 미커버 라인은 참고용 보고서로만 활용합니다.

### 9단계: 최종 결과 요약

- 모든 테스트가 성공적으로 해결되었음을 알리고, 구현된 소스 파일 목록과 최종 테스트 통과 수치 및 커버리지 통계를 깔끔하게 요약하여 개발자에게 공유합니다.
