# TDD Red 테스트 코드 작성 스킬 (tdd-red)

이 스킬은 확정된 명세와 테스트 시나리오를 바탕으로, 실제 비즈니스 로직을 구현하기 전에 **실패하는 테스트 코드(Red)**를 안전하게 작성하고 실행하여 TDD(Test Driven Development) 개발의 시작 단계를 보장하는 가이드라인을 강제합니다.

## 사용법

- `/tdd-red [GitHub 이슈 번호]` (예: `/tdd-red 1`)

## 입력 인자

- `$ARGUMENTS`: GitHub 이슈 번호 (예: `1`)

## 상세 실행 단계

이 스킬이 활성화되면 다음 단계를 순서대로 엄격히 수행합니다.

### 1단계: 시나리오 및 시그니처 읽기

- **동작**: `docs/features/{tag}/issue-{번호}.md` 파일에서 해당 이슈(`$ARGUMENTS`)의 확정된 함수/컴포넌트 시그니처와 정상/경계/예외 테스트 시나리오 목록을 읽어옵니다. (여기서 `{번호}`는 `$ARGUMENTS` 값이며, `{tag}`는 현재 피처 디렉토리 이름입니다)
- **분석**: 작성해야 하는 테스트 대상 파일의 정확한 위치와 대상 함수/컴포넌트를 명확히 식별합니다.

### 2단계: 실패하는 테스트 코드 작성 및 순차 실행

- **환경**: **Vitest**와 **React Testing Library**를 기본 테스트 프레임워크로 사용합니다.
- **테스트 파일 명명 및 위치 컨벤션**:
  - 테스트 대상 파일과 **동일한 디렉토리**에 테스트 파일을 생성합니다.
  - 파일 이름은 `{대상파일명}.test.ts` 또는 `{대상파일명}.test.tsx` 형식을 따릅니다.
    - 예: `src/api/tags.ts` → `src/api/tags.test.ts`
    - 예: `src/components/TagInput.tsx` → `src/components/TagInput.test.tsx`
- **구조화**:
  - `describe` 블록을 활용하여 테스트 대상 함수 또는 컴포넌트 단위로 테스트 케이스를 그룹화합니다.
    - 예: `describe('addTag', () => { ... })`
  - 개별 테스트 케이스(`it` 또는 `test`)의 이름은 반드시 **`should [기대 동작] when [조건]`** 형식을 준수하여 작성합니다.
- **TDD Red 구현 순서**:
  - **제약 사항: `src/` 디렉토리 하위의 비즈니스 구현 코드는 절대 생성하거나 수정하지 않습니다.** (구현이 아예 비어있거나 타입 선언만 존재하는 상태여야 합니다)
  - 도출된 시나리오를 **한 번에 하나씩** 테스트 코드로 작성합니다.
  - 하나의 테스트를 작성할 때마다 즉시 실행(예: `npx vitest run <테스트파일명>`)하여 **실패(Red)하는 것을 명확히 확인**합니다.
  - 해당 테스트의 실패를 확인한 후에만 다음 시나리오의 테스트 코드를 작성하며 순차적으로 나아갑니다.

### 3단계: 전체 테스트 실행 및 검사

- 모든 시나리오에 대한 테스트 케이스 작성이 끝나면, 전체 테스트 명령어(`npm test` 또는 `npx vitest run`)를 실행합니다.
- 작성된 모든 테스트가 예외 없이 **실패(Red)** 상태를 유지하고 있는지 최종적으로 확인합니다.
- 전체 테스트 실패 결과를 로그와 함께 개발자에게 요약하여 공유하고 단계를 마무리합니다.
