# TDD 풀 사이클 오케스트레이터 스킬 (tdd-loop)

이 스킬은 이슈 번호를 입력받아 `CLAUDE.md`에 명시된 TDD 이슈 사이클의 0단계부터 8단계까지의 과정을 체계적으로 순서 보장하여 구동해주는 오케스트레이터 지침입니다.

## 사용법

- `/tdd-loop [GitHub 이슈 번호]` (예: `/tdd-loop 1`)

## 입력 인자

- `$ARGUMENTS`: GitHub 이슈 번호 (예: `1`)

---

## 📌 전체 워크플로우 단계 (TDD 이슈 사이클)

이 스킬이 가동되면 다음 순서에 따라 단계별로 하위 스킬 및 에이전트를 차례대로 구동합니다.
컨테이너 자체는 각 단계 사이에 불필요한 게이트("다음 단계로 진행할까요?")를 만들지 않고, 하위 스킬의 승인 게이트 완료 여부만 확인하여 즉시 다음 단계로 전이합니다.

---

### 0단계: 자식 브랜치 생성 (사전 점검)

1. **이슈 및 AC 조회**: `gh issue view $ARGUMENTS`를 수행하여 해당 이슈의 제목, 본문, Acceptance Criteria(AC) 목록을 확인하고 이를 내부 컨텍스트에 저장합니다.
2. **작업 공간 상태 검사**: `git status`를 실행합니다. 커밋되지 않은 변경 사항(Uncommitted changes)이 존재하면 안전을 위해 **즉시 동작을 중단**하고 사용자에게 알립니다.
3. **부모 브랜치 검사 및 푸시**:
   - 현재 활성화된 부모 브랜치를 로컬과 원격의 싱크를 맞추기 위해 원격에 먼저 push(`git push origin <feature>`)합니다.
4. **자식 브랜치 분기 처리**:
   - 부모 feature 브랜치로부터 `git checkout -b <feature>-issue-N` 명령어를 통해 자식 브랜치를 파생하고 전환합니다. (부모명을 `/`로 prefix하면 ref 충돌이 날 수 있으므로 피합니다)
   - **동일 브랜치 존재 시**: 만약 로컬에 이미 동일한 자식 브랜치가 존재한다면, 사용자에게 **"기존 브랜치를 삭제하고 다시 분기(덮어쓰기)할지"** 아니면 **"취소할지"** 응답을 확인하여 처리합니다.
5. 사전 점검 단계가 하나라도 실패하면 구체적인 사유를 보고하고 **전체 과정을 즉시 중단**합니다.

---

### 1단계: 시나리오 및 시그니처 도출

- **호출**: `/test-scenarios $ARGUMENTS`
- **목적**: 함수 및 컴포넌트의 시그니처(이름, 타입, 에러 케이스 등)를 확정하고, 정상/경계/예외 테스트 시나리오를 도출하여 `issue-{N}.md` 상단에 기록합니다.
- **게이트**: 명세 및 시나리오 검토/승인 게이트가 그대로 작동합니다. 승인 완료 시 다음 단계로 자동 전이합니다.

---

### 2단계: 실패 테스트 작성 (TDD Red)

- **호출**: `/tdd-red $ARGUMENTS`
- **목적**: 이전 단계의 시나리오를 바탕으로 실패(Red)하는 테스트 코드를 하나씩 순차적으로 작성하고 실행합니다.
- **검증**: 만약 테스트 러너가 신규 테스트 파일을 정상 로드하지 못하는(Collect 실패) 경우, 개발자에게 스텁(Stub) 파일 생성 절차를 올바르게 안내하도록 조치합니다.

---

### 3단계: 최소 구현 및 통과 (TDD Green)

- **호출**: `/tdd-green $ARGUMENTS`
- **목적**: 실패하는 테스트들을 통과시키는 데 꼭 필요한 **최소한의 구현 코드**만 작성하여 기능 명세를 완전히 완성합니다.
- **루프**: 최대 5회의 피드백 루프(수정 및 테스트 재검증) 및 회귀 오류 감시 절차가 정상 구동되는지 확인합니다.

---

### 4단계: AC 만족성 교차 검증 (@ac-verifier)

- **호출**: `@ac-verifier $ARGUMENTS` 에피소드 가동
- **목적**: 구현된 로직이 단순히 테스트 파일만 통과한 것인지, 아니면 실제 이슈의 본래 목적(Acceptance Criteria)을 완벽하게 만족하고 있는지 독립적으로 검출합니다.
- **예외**: 만약 요구사항과의 괴리(Gap)가 발견되면, 검증 요약 보고서를 즉시 작성하여 사용자에게 리포트하고 수정을 유도합니다.

---

### 5단계: 구조 개선 및 리팩토링 (TDD Refactor)

- **호출**: `/tdd-refactor $ARGUMENTS`
- **목적**: 비즈니스 로직에 추가적인 부작용을 일으키지 않고 테스트 통과 상태(`Green`)를 보존하며 소스 코드 구조를 개선합니다.
- **규칙**: 매 리팩토링 변경 단위마다 `npm test`가 자동으로 실행되어 회귀 현상을 방지하고, 에러 회귀가 발생하면 즉각 롤백합니다.

---

### 6단계: 타입 및 보안성 검사 (Security Review)

- **호출**: `/security-review $ARGUMENTS`
- **목적**: 리팩토링 완료 후 타입 에러(`tsc`), 서드파티 패키지 보안 취약성(`npm audit`), 그리고 소스 내 민감 환경 변수 노출 여부를 점검합니다.
- **분류**: 수집된 경고 요소를 3등급(즉시수정, 권장수정, 무시가능)으로 체계화하여 해결을 마칩니다.

---

### 7단계: 커밋 (commit)

- **목적**: 리팩토링 및 타입/보안 스캔이 통과하면, 관심사별로 분리하여 커밋합니다.
- **규칙**: 커밋 메시지는 conventional commit 표준 포맷(feat, fix, refactor 등 접두사 및 소문자 기입 규칙)에 완벽히 부합해야 합니다.

---

### 8단계: 풀 리퀘스트 생성 및 이슈 종결 (PR)

- **호출**: `/create-pr $ARGUMENTS`
- **목적**: 작업된 변경 사항의 브랜치를 원격에 푸시하고 PR을 생성하여 이슈 종결 프로세스를 가동합니다.
- **명세**:
  - PR의 Base 브랜치는 사전 점검 단계의 부모 브랜치인 `feature/<spec>`으로 명확하게 타겟팅합니다.
  - PR 본문(Body)에 자동으로 `"Closes #$ARGUMENTS"`를 명시하여 main 머지 시 이슈가 정상 종결될 수 있도록 연결합니다.
  - 생성 완료된 PR 링크를 GitHub 이슈의 코멘트로 추가해 공유합니다.

---

## ⚠️ 실행 제약 및 특징

1. **중단 지점의 명확성**: 전체 루프 수행 도중 특정 단계가 에러로 실패하거나, 사용자의 거절로 중단될 경우, 컨테이너는 **"어느 단계(예: 3단계 TDD Green)의 어떤 에러로 인해 멈추었는지"** 명확하게 원인을 추적하여 에러 메시지를 표시하고 실행을 중단해야 합니다.
2. **이중 질문 배제**: 각 하위 단계 스킬들이 자체적으로 가지고 있는 승인 프롬프트(예: 시그니처 승인, 리팩토링 승인 등)는 그대로 사용자에게 전달되어 응답을 기다려야 합니다. 단, 컨테이너(tdd-loop)가 추가적으로 "다음 단계로 넘어갈까요?"와 같은 불필요한 확인 단계를 덧붙여 사용자에게 피로감을 주지 않도록 하십시오.
