# TDD Refactor 코드 구조 개선 스킬 (tdd-refactor)

이 스킬은 기존에 성공(`Green`)한 테스트들의 통과 상태를 엄격히 보장하면서, 구현된 코드베이스의 가독성, 유지보수성, 성능을 체계적으로 개선하는 TDD 리팩토링 프로세스를 강제합니다.

## 사용법

- `/tdd-refactor [GitHub 이슈 번호]` (예: `/tdd-refactor 1`)

## 입력 인자

- `$ARGUMENTS`: GitHub 이슈 번호 (예: `1`)

## 상세 실행 단계

이 스킬이 활성화되면 다음 단계를 순서대로 엄격히 수행하며, 개발자의 승인과 테스트 회귀 여부를 끊임없이 점검합니다.

### 1단계: 프로젝트 컨벤션 확인

- **동작**: 프로젝트 루트의 `CLAUDE.md` 파일을 읽어 프로젝트별 고유 코딩 컨벤션(네이밍 스타일, 아키텍처 패턴, 상태 관리 방식, 스타일링 규칙 등)을 정확히 파악합니다.

### 2단계: 사전 상태 검증 (Green 확인)

- **동작**: 리팩토링을 시작하기 전, `npm test`를 실행하여 현재 모든 테스트 케이스가 예외 없이 정상 통과하는 안정한 상태(`Green`)인지 확인합니다. 만약 실패하는 테스트가 있다면 리팩토링을 시작하지 않고 먼저 수정해야 합니다.

### 3단계: 리팩토링 대상 파일 식별 및 검토

- **추출**:
  - `git diff main...HEAD --name-only -- src/` 명령을 실행하여 현재 작업 브랜치에서 수정/추가된 `src/` 하위 소스 파일들의 목록을 가져옵니다.
  - 이 중 테스트 파일(`*.test.ts`, `*.test.tsx`)을 제외한 핵심 소스 코드 파일들만 식별합니다.
- **범용 점검 기준에 따른 검토**:
  - 식별된 각 소스 파일에 대해 아래 5가지 기준을 적용하여 리팩토링이 필요한 부분을 꼼꼼히 찾아냅니다.
    1. **중복 제거**: 동일하거나 매우 유사한 비즈니스 로직, 렌더링 패턴이 반복적으로 등장하는가?
    2. **네이밍 명확성**: 변수, 함수, 컴포넌트, 인터페이스의 이름이 그 역할과 의도를 명확하게 대변하는가?
    3. **단일 책임 (SRP)**: 하나의 함수나 컴포넌트가 너무 복잡하게 다중 역할을 처리하고 있는가? (필요시 하브 모듈이나 하위 컴포넌트로 분리)
    4. **불필요한 복잡도 제거**: 의미를 알 수 없는 매직 넘버/스트링이 존재하는가? 조건문 중첩이 과도한가? 불필요하게 복잡한 디자인 패턴이 사용되었는가?
    5. **CLAUDE.md 불일치**: 1단계에서 확인한 프로젝트 표준 패턴 및 디자인 스펙과 어긋난 구조가 존재하는가?

### 4단계: 리팩토링 계획 보고 및 승인 대기

- 3단계 분석 결과를 토대로, 리팩토링이 필요한 파일 및 구체적인 리팩토링 계획(목표 및 예상 변경 내용)을 정리하여 개발자에게 상세히 보고합니다.
- 개발자가 계획을 확인하고 **승인을 내릴 때까지 작업을 멈추고 대기**합니다.

### 5단계: 미세 단위 점진적 수정 및 테스트 검증

- 개발자의 승인을 득하면 리팩토링을 시작합니다.
- **마이크로 리팩토링 규칙**:
  - **단 한 번에 하나의 변경 사항만** 코드로 작성합니다.
  - 하나의 리팩토링이 끝날 때마다 즉시 `npm test`를 실행하여 전체 테스트가 여전히 모두 성공적으로 동작하는지 확인합니다.
  - 검증에 성공하면 다음 변경 단계로 진행합니다.

### 6단계: 에러 회귀 발생 시 즉각 롤백

- 리팩토링 수정 직후 테스트가 하나라도 실패(`Red`)하는 경우, 독단적으로 디버깅하며 코드를 덧붙이지 말고 **즉시 해당 변경을 롤백(Rollback)**하여 이전의 안정적인 통과 상태로 되돌립니다.
- 그 후 원인을 원점에서 재분석하여 다른 리팩토링 접근 방식을 모색합니다.

### 7단계: 결과 요약 및 보고

- 모든 리팩토링 작업이 끝나고 전체 테스트가 통과함을 확인한 뒤, 리팩토링 전후의 구조 개선 세부 내용(예: 중복 코드 제거 결과, 가독성 개선 사항 등)을 요약하여 개발자에게 공유합니다.

## 제약 사항

- 리팩토링 단계에서는 **새로운 기능을 절대 추가하지 않습니다.**
- 테스트 파일(`*.test.ts`, `*.test.tsx`)을 수정하거나 변경해서는 안 됩니다.
- `src/` 디렉토리 외부의 파일은 절대 수정하지 않습니다.
- 3단계에서 `git diff`를 통해 확인된 수정 범위 이외의 소스 파일들을 수정해서는 안 됩니다.
